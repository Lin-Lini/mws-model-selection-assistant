# Ассистент выбора моделей MWS GPT Model Hub

Проект реализует сервис-ассистент для подбора моделей из **MWS GPT Model Hub** под продуктовый сценарий, требования к качеству и бюджет. Сервис принимает описание задачи через OpenAI-совместимый API, получает актуальные данные о моделях и тарифах из документации MWS во время обработки запроса, рассчитывает примерную стоимость использования и возвращает структурированный отчёт с рекомендациями.

Решение выполнено на Python и использует **Google ADK** как основу агентного сценария. HTTP-слой сделан без полноценного web-фреймворка, чтобы не выходить за ограничения тестового задания.

## Что реализовано по заданию

В проекте закрыты все требования тестового задания:

- OpenAI-совместимый эндпоинт `POST /v1/chat/completions`;
- агентный сценарий на базе Google ADK;
- динамическая загрузка каталога моделей, тарифов и лимитов MWS во время запроса;
- отсутствие захардкоженных рабочих цен в рабочем сценарии;
- in-memory кеш данных MWS в пределах запуска сервиса и пользовательской сессии;
- расчёт примерной месячной стоимости по входящим и исходящим токенам;
- структурированный ответ с блоками:
  - `Входные данные`;
  - `Рекомендованные модели`;
  - `Расчеты`;
  - `Пояснения и ограничения`;
- поддержка диалога в рамках одной сессии через заголовок `X-Session-Id`;
- локальный запуск через `venv` и через `docker compose`;
- потоковые ответы при `stream=true`;
- автотесты для парсинга, расчётов, рекомендаций, API и защитных сценариев;
- observability: healthcheck, метрики и логирование ключевых шагов.

## Архитектурный подход

Выбран последовательный агентный сценарий: **SequentialAgent workflow**. Такой вариант подходит для задачи, потому что подбор модели удобно разложить на понятные этапы: сначала разобрать ввод пользователя, затем получить каталог, затем рассчитать стоимость и ранжировать модели, после этого сформировать отчёт.

Важно, что в проекте не используется LLM для «угадывания» тарифов или выбора модели по скрытым эвристикам. Расчёты, фильтрация и ранжирование выполнены детерминированно. Google ADK используется как слой оркестрации агентного процесса, а бизнес-логика остаётся проверяемой и покрытой тестами.

### Роли агентов

1. **IntakeAgent**
   - читает сообщения из OpenAI-совместимого запроса;
   - извлекает из текста или JSON структуру сценария: тип задачи, трафик, токены, бюджет, приоритет;
   - объединяет новые вводные с ранее сохранёнными данными той же сессии.

2. **CatalogAgent**
   - загружает страницы документации MWS по моделям, тарифам и лимитам;
   - парсит HTML-страницы в нормализованный внутренний каталог;
   - использует in-memory кеш, чтобы не делать лишние запросы к MWS в рамках одной сессии.

3. **RecommendationAgent**
   - применяет обязательные фильтры: тип задачи, поддержка изображений, режим эмбеддингов, контекстное окно;
   - рассчитывает примерную стоимость для подходящих моделей;
   - ранжирует кандидатов с учётом бюджета, приоритета пользователя и характеристик модели.

4. **ReportAgent**
   - формирует финальный текстовый отчёт;
   - возвращает его в формате ответа OpenAI Chat Completions API.

## Схема выполнения запроса

```text
Клиент
  -> POST /v1/chat/completions
  -> HTTP handler на базе http.server
  -> ADKAssistantRuntime
  -> SequentialAgent
       -> IntakeAgent
       -> CatalogAgent
       -> RecommendationAgent
       -> ReportAgent
  -> OpenAI-совместимый JSON-ответ или SSE stream
```

## Источники данных

В рабочем режиме сервис динамически обращается к документации MWS:

- `https://mws.ru/docs/cloud-platform/gpt/general/gpt-models.html` — каталог моделей;
- `https://mws.ru/docs/cloud-platform/gpt/general/pricing.html` — тарифы и правила тарификации;
- `https://mws.ru/docs/cloud-platform/gpt/general/quotas-limits.html` — квоты и лимиты.

Цены для рабочего сценария не зашиты в код. Они извлекаются из документации MWS во время запроса. Это принципиальная часть решения, потому что тарифы могут измениться, а тестовое задание прямо запрещает хардкодить прайсы или получать их через LLM.

Для тестов и офлайн-демонстрации можно использовать локальные фикстуры:

```bash
MWS_FIXTURE_DIR=tests/fixtures/mws python -m app.server
```

## Логика подбора модели

Ранжирование сделано детерминированным и объяснимым. Сервис не просто выбирает первую модель из таблицы, а применяет набор фильтров и скоринговых правил.

### Жёсткие фильтры

- Для обычных генеративных сценариев исключаются модели эмбеддингов.
- Для задач построения эмбеддингов исключаются генеративные модели.
- Если пользователю нужен ввод изображений, остаются только модели с поддержкой изображений.
- Если средний размер запроса больше контекстного окна модели, модель получает сильный штраф.
- Если указан бюджет, рассчитанная стоимость сравнивается с месячным ограничением.

### Мягкое ранжирование

На итоговую оценку влияют:

- соответствие типу задачи;
- приоритет пользователя: качество, баланс, экономичность или скорость;
- запас по контекстному окну;
- попадание в бюджет;
- относительная стоимость;
- пригодность модели для задач программирования, если выбран сценарий программирования.

## Расчёт стоимости

Стоимость не считается по наивной формуле `цена × общее число токенов`. В расчёте учитываются особенности тарификации MWS:

- отпускная единица тарификации;
- накопление остатка токенов в течение 24 часов;
- промо-тарифы, если они активны на текущую дату;
- отдельные цены для входящих и исходящих токенов.

В отчёте выводятся две оценки:

1. **Оценка за 24 часа** — предполагает, что запросы в течение суток накапливаются до кратности отпускной единице.
2. **Минимальная оценка при редких запросах** — нижняя граница для разреженного трафика, когда остатки токенов могут сгорать через 24 часа.

Обе оценки являются приближенными, потому что реальная токенизация зависит от конкретной модели и содержимого запросов.

## API

Основной эндпоинт:

```text
POST /v1/chat/completions
```

Формат запроса совместим с OpenAI Chat Completions API. Ответ также возвращается в формате Chat Completions: итоговый отчёт находится в `choices[0].message.content`.

Дополнительные эндпоинты:

```text
GET /healthz   Проверка работоспособности сервиса
GET /metrics   Простые runtime-метрики
```

## Переменные окружения

| Переменная | Значение по умолчанию | Назначение |
|---|---:|---|
| `HOST` | `0.0.0.0` | Адрес, на котором слушает сервер |
| `PORT` | `8000` | Порт внутри приложения |
| `HOST_PORT` | `8000` | Порт на хосте для Docker Compose |
| `MAX_BODY_BYTES` | `1048576` | Максимальный размер тела запроса |
| `SESSION_TTL_SECONDS` | `7200` | Время жизни in-memory кеша и сессии |
| `REQUEST_TIMEOUT_SECONDS` | `15` | Таймаут HTTP-запросов к MWS |
| `MWS_FIXTURE_DIR` | пусто | Путь к локальным HTML-фикстурам MWS |
| `LOG_LEVEL` | `INFO` | Уровень логирования |

## Локальный запуск через venv

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements-dev.txt
python -m app.server
```

После запуска API доступен по адресу:

```text
http://localhost:8000
```

## Запуск через Docker

Сборка и запуск сервиса:

```bash
docker compose up --build assistant
```

Если порт `8000` на хосте занят, можно выбрать другой порт:

```bash
HOST_PORT=8010 docker compose up --build assistant
```

В этом случае API будет доступен по адресу:

```text
http://localhost:8010
```

Остановка контейнеров:

```bash
docker compose down --remove-orphans
```

## Проверка через Docker

Полная проверка одной командой:

```bash
./scripts/docker_check.sh
```

Скрипт выполняет полный цикл:

1. собирает Docker-образ;
2. запускает `pytest -q` внутри контейнера;
3. поднимает сервис `assistant`;
4. ждёт успешный `/healthz`;
5. запускает `scripts/pentest_runner.py` из отдельного контейнера;
6. выводит последние логи сервиса;
7. очищает контейнеры после проверки.

Отдельный запуск тестов в Docker:

```bash
docker compose run --rm tests
```

Отдельный запуск pentest-проверки против Docker-сервиса:

```bash
docker compose up -d assistant
docker compose run --rm pentest
docker compose down --remove-orphans
```

## Пример формы ответа

Сервис возвращает стандартный объект Chat Completions. Сокращённый пример:

```json
{
  "id": "chatcmpl-...",
  "object": "chat.completion",
  "created": 1710000000,
  "model": "mws-model-selector",
  "choices": [
    {
      "index": 0,
      "finish_reason": "stop",
      "message": {
        "role": "assistant",
        "content": "Входные данные\n- Тип кейса: чат-ассистент\n- Трафик: 1 500 запросов в день\n- Средний вход: 1 200 токенов/запрос\n- Средний выход: 500 токенов/запрос\n- Бюджет: 25 000.00 ₽ / месяц\n\nРекомендованные модели\n1. qwen3-32b | разработчик=QWEN | вход=текст | выход=текст | контекст=40k\n   - Подходит для обычного текстового диалога без режима эмбеддингов.\n   - Оценка месячной стоимости укладывается в бюджет.\n\nРасчеты\n- qwen3-32b: оценка за 24 часа=..., минимальная оценка при редких запросах=...\n  тарифы: вход=... ₽/1000 токенов, выход=... ₽/1000 токенов, единица тарификации=100\n\nПояснения и ограничения\n- Источник цен и списка моделей: документация MWS, полученная во время запроса.\n- Оценка за 24 часа предполагает накопление токенов в пределах суток до кратности отпускной единице."
      }
    }
  ],
  "usage": {
    "prompt_tokens": 0,
    "completion_tokens": 1,
    "total_tokens": 1
  }
}
```

Порядок моделей и конкретные тарифы могут измениться, потому что сервис каждый раз работает с актуальной документацией MWS. Это ожидаемое поведение: рабочих цены не должны быть зашиты в репозиторий.

## Пример диалога в одной сессии

Первый запрос:

```json
{
  "model": "mws-model-selector",
  "messages": [
    {
      "role": "user",
      "content": "Нужен чат-ассистент поддержки. 1000 запросов в день. 1000 входящих и 300 исходящих токенов."
    }
  ]
}
```

Второй запрос с тем же заголовком `X-Session-Id`:

```json
{
  "model": "mws-model-selector",
  "messages": [
    {
      "role": "user",
      "content": "Бюджет до 7000 ₽ в месяц, важнее экономичность."
    }
  ]
}
```

Во втором запросе сервис использует ранее сохранённый сценарий трафика и дополняет его новым бюджетом и приоритетом.
